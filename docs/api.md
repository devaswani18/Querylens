# QueryLens AI — API Contract (V1)

Refer to `docs/requirements.md` for scope and `docs/architecture.md` for the service layer that implements these. This document is the exact, binding contract every endpoint must follow. Windsurf prompts for backend routes should reference this file directly.

## Response Envelope (applies to every endpoint, no exceptions)

**Success:**
```json
{
  "success": true,
  "data": { ... }
}
```

**Failure:**
```json
{
  "success": false,
  "error": {
    "code": "STRING_ERROR_CODE",
    "message": "Human-readable message safe to show in the UI"
  }
}
```

No endpoint returns raw arrays/objects at the top level. No endpoint leaks stack traces, raw SQL error text, or internal file paths in `error.message`.

## Standard Error Codes (used across endpoints)
| Code | HTTP Status | Meaning |
|---|---|---|
| `INVALID_INPUT` | 400 | Request body failed validation |
| `QUERY_REJECTED` | 400 | SQL contains a disallowed statement (DELETE/DROP/UPDATE/etc.) |
| `QUERY_TIMEOUT` | 408 | Query exceeded the 10s execution limit |
| `ROW_LIMIT_EXCEEDED` | 200 | Query succeeded but was capped at 500 rows — not an error, included as a warning flag in `data` instead (see below) |
| `NOT_FOUND` | 404 | Requested table/resource doesn't exist |
| `INDEX_STATEMENT_MISMATCH` | 403 | `/apply-index` received a statement not previously generated by the rule engine this session |
| `GEMINI_UNAVAILABLE` | 502 | Gemini API call failed or rate-limited |
| `DB_ERROR` | 500 | Unexpected database error (message sanitized before returning) |
| `RATE_LIMITED` | 429 | Client hit the Express rate limiter |

## 1. Execute Query
`POST /api/query/execute`

**Request:**
```json
{ "sql": "SELECT * FROM orders WHERE customer_id = 42;" }
```

**Success (200):**
```json
{
  "success": true,
  "data": {
    "rows": [ { "id": 1, "customer_id": 42, "...": "..." } ],
    "rowCount": 12,
    "executionTimeMs": 461,
    "truncated": false
  }
}
```
`truncated: true` when the result hit the 500-row cap — this is how `ROW_LIMIT_EXCEEDED` surfaces; it is not an error state.

**Failure examples:**
- Disallowed statement → 400, `QUERY_REJECTED`
- Timeout → 408, `QUERY_TIMEOUT`

## 2. Schema Explorer

`GET /api/schema/tables`

**Success (200):**
```json
{ "success": true, "data": { "tables": ["users", "products", "orders", "payments", "reviews"] } }
```

`GET /api/schema/tables/:name`

**Success (200):**
```json
{
  "success": true,
  "data": {
    "table": "orders",
    "columns": [
      { "name": "id", "type": "integer", "isPrimaryKey": true, "isForeignKey": false },
      { "name": "customer_id", "type": "integer", "isPrimaryKey": false, "isForeignKey": true, "referencesTable": "users", "referencesColumn": "id" }
    ],
    "indexes": [
      { "name": "orders_pkey", "columns": ["id"], "isUnique": true }
    ]
  }
}
```

**Failure:** unknown table name → 404, `NOT_FOUND`

## 3. Natural Language → SQL
`POST /api/nl-to-sql`

**Request:**
```json
{ "prompt": "Show the 10 most recent orders for customer 42" }
```

**Success (200):**
```json
{ "success": true, "data": { "sql": "SELECT * FROM orders WHERE customer_id = 42 ORDER BY ordered_at DESC LIMIT 10;" } }
```
Frontend places this in the Monaco editor for review. **Never auto-executed by this endpoint.**

**Failure:** Gemini call fails → 502, `GEMINI_UNAVAILABLE`

## 4. Explain (core feature)
`POST /api/explain`

**Request:**
```json
{ "sql": "SELECT * FROM orders WHERE customer_id = 42;" }
```

**Success (200):**
```json
{
  "success": true,
  "data": {
    "rawPlan": { "...": "raw EXPLAIN ANALYZE JSON, kept for the collapsible 'view raw plan' UI option" },
    "findings": {
      "issues": [
        { "type": "SEQ_SCAN", "table": "orders", "rows": 150000, "costEstimate": 3204.5, "actualTimeMs": 461 }
      ],
      "suggestedIndexes": [
        {
          "table": "orders",
          "column": "customer_id",
          "statement": "CREATE INDEX idx_orders_customer ON orders(customer_id);"
        }
      ]
    },
    "explanation": "PostgreSQL scanned all 150,000 rows in the orders table because there is no index on customer_id. Adding an index lets it jump directly to matching rows instead of checking every one."
  }
}
```
`findings` comes from the deterministic rule engine (`architecture.md` → `ruleEngine.ts`). `explanation` is the only field generated by Gemini — it explains `findings`, it does not decide them.

**Failure:** SQL rejected by validator (only SELECT is explainable) → 400, `QUERY_REJECTED`; Gemini fails → 502, `GEMINI_UNAVAILABLE` (in this case, still return `findings` and `rawPlan` with `explanation: null` rather than failing the whole request — the rule-based findings remain useful without the AI explanation)

## 5. Apply Suggested Index
`POST /api/apply-index`

**Request:**
```json
{ "statement": "CREATE INDEX idx_orders_customer ON orders(customer_id);" }
```
Server validates this exact string was returned by `/api/explain` in `suggestedIndexes` during the current session (tracked server-side, e.g. in-memory session map keyed by a session token — not trusted from the client alone).

**Success (200):**
```json
{ "success": true, "data": { "applied": true, "statement": "CREATE INDEX idx_orders_customer ON orders(customer_id);" } }
```

**Failure:** statement doesn't match a known suggestion → 403, `INDEX_STATEMENT_MISMATCH`

Frontend flow after success: automatically re-run the original query via `/api/query/execute` and display the before/after `executionTimeMs` comparison.

---

## Rate Limiting
Applied globally via `express-rate-limit`: 100 requests / 15 minutes per IP for V1 (single-user local demo tool — generous limit, just a safety net, not a monetization gate). Exceeding it returns 429, `RATE_LIMITED`.

## CORS
Backend allows only the deployed frontend origin (and `localhost:*` in development) — configured via `helmet`/`cors` in `app.ts`, origin read from an env var, never wildcarded in production.
